Bottom-Up parsing

  • También llamado shift-reduce parsing.
  • Método de parseo más general (reconoce un conjunto más amplio de lenguajes) que top-down.
  • Intenta construir un árbol de parseo comenzando desde las hojas del árbol, construyendo hasta llegar a la raíz.
  • Se puede visualizar este proceso como reducir el string de entrada hasta obtener el símbolo inicial de la gramática.
  • En cada paso de reducción se reemplaza un substring del input por el no terminal del lado izquierdo de alguna producción.
  • Al finalizar este proceso, se obtiene un arbol por una derivación derecha.

Bottom-Up parsing

Ejemplo

S -> aABe
A -> Abc | b
B -> d
  • El string abbcde puede reducirse hasta S de la siguiente forma:
    • abbcde
    • aAbcde
    • aAbcde
    • aAde
    • aAde
    • aABe
    • aABe
    • S

Bottom-Up parsing

Ejemplo

S -> aABe
A -> Abc | b
B -> d
Las reducciones anteriores reflejan la siguiente derivación derecha:

S -> aABe -> aAde -> aAbcde -> abbcde

Bottom-Up parsing

  • Utiliza derivación derecha.
  • No necesita factorizar las gramáticas por la izquierda.
  • Podemos utilizar gramáticas más 'naturales' o fáciles de entender.

Shift-Reduce Parsing

Bottom-Up parsing

Shift-Reduce Parsing

Acciones

  • Shift: Insertar el token actual del input al tope del stack (push), mover el cursor al siguiente token.
  • Reduce: El tope del stack debe hacer match con el RHS de alguna producción.
    • Expulsar del stack dichos símbolos (pop).
    • Insertar al stack el no-terminal del LHS de la producción.
  • Accept: El stack debe contener únicamente el símbolo de inicio, debimos haber recorrido todo el input.
  • Error: Recorrimos todo el input, pero el stack tiene más que el símbolo de inicio. Rechaza el input.

Shift-Reduce Parsing

Funcionamiento

  • En teoría es muy simple, el stack y el token actual determinan la acción a tomar.
  • En la práctica no resulta tan fácil:
    • ¿Qué sucede si el stack hace match con el RHS de varias producciones?
    • ¿Es necesario reduchir cada vez que el stack contiene un RHS válido?

Shift-Reduce Parsing

Conflictos

  • Reduce/Reduce: Si existen varias producciones por las cuales reducir.
  • Shift/Reduce: El parser no sabe si hacer una reducción o un shift, refleja una gramática ambigua.

LR(k) Parsing

Bottom-up parsing

LR(k) parsing

  • Implementación de Bottom-Up parsing.
  • L: Left to right scanning.
  • R: Rightmost derivation.
  • k: Lookahead tokens..

  • Muy poderoso, puede reconocer prácticamente cualquier lenguaje de programación descrito por una CFG.
  • Implementación genérica sin backtracking, altamente eficiente.
  • Capaz de reconocer más lenguajes que los parsers predictivos.

  • Muy laborioso de implementar a mano.
  • Afortunadamente existen herramientas automatizadas.

Implementación

LR(0)

LR(0) parsing

Implementación

  • Agregar S' -> S$.
  • Calcular conjunto de items.
  • Construir DFA.
  • Calcular tabla de parseo en base al DFA.

LR(0) parsing

Implementación

  • Debemos aumentar la gramática original agregando la producción S' -> S$.
  • Donde S es el símbolo de inicio.
  • Esto se hace con el motivo de que el parser sepa que debe detenerse al reducir esta producción.

Items

LR(0) parsing

  • Permite identificar cuánto se ha escaneado sobre una producción.
  • Para eso se utiliza un punto ⋅.
  • Ejemplo: A -> XYZ
    • 	    A -> ⋅XYZ
	    A -> X⋅YZ
	    A -> XY⋅XZ
	    A -> XYZ⋅
  • X -> ε sólo produce el item X -> ⋅

Items

Ejemplo

  • S -> X$
  • X -> (X)
  • X -> ( )
  • 	      S -> ⋅X$
	      S -> X⋅$
	      X -> ⋅(X)
	      X -> (⋅X)
	      X -> (X⋅)
	      X -> (X)⋅
	      X -> ⋅()
	      X -> (⋅)
	      X -> ()⋅

DFA

LR(0) parsing

  • Construído en base a los items.
  • Cada estado del DFA será un subjunto de items.
  • Si un estado contiene el item A -> α ⋅ c β
    • Y el siguiente token es c.
    • Debe hacer un shift, el siguiente estado contendrá A -> αc⋅β
  • Si un estado contiene el item A -> α ⋅
    • Significa que el parser ya consumió toda la producción.
    • Debe hacer una reducción utilizando A -> α.
  • Si un estado contiene el item S -> α ⋅ $
    • Y además ya se consumió todo el string de entrada.
    • Significa que el parser logró reducir hasta el símbolo de inicio.
    • Debe aceptar el string como parte del lenguaje.

DFA

LR(0) parsing

  • Para construir el DFA necesitamos 2 funciones.
  • Closure
    • Obtiene todos los items en el mismo estado.
  • Goto
    • Obtiene el siguiente estado, dado un estado y un símbolo de la gramática.

Closure

LR(0) parsing

  • Para un conjunto de items I:

  • Todo item contenido en I, pertenece a Closure(I)

  • Si A -> α ⋅ B β ∈ Closure(I) y B -> ⋅ γ
  • Entonces B -> ⋅ γ ∈ Closure(I)

  • Repetir hasta que no se puedan agregar más items a Closure(I)

Closure

Ejemplo

  • closure({X -> (⋅X)}):
    • X -> (⋅X)
    • X -> ⋅ (X)
    • X -> ⋅ ( )

Closure

Ejemplo

  • closure({S -> ⋅X$}):
    • S -> ⋅ X$
    • X -> ⋅ (X)
    • X -> ⋅ ( )

Goto

LR(0) parsing

  • Para un conjunto de items I.
  • Y un símbolo de la gramática X.
  • goto(I, X) = closure({A -> α X ⋅ β | A -> ⋅ X β ∈ I})

  • En otras palabras, si A -> ⋅ X β ∈ I, buscamos los items resultantes al mover el punto sobre X.

Goto

Ejemplo

  • goto({X -> ( ⋅ X)}, X):
    • X -> (X ⋅ )

Goto

Ejemplo

  • goto({X -> ⋅ (X)}, ( ) :
    • X -> ( ⋅ X)
    • X -> ⋅ (X)
    • X -> ⋅ ( )

DFA

Algoritmo

  • Iniciar con la producción S -> ⋅ β $
  • El primer estado será closure({S -> ⋅ β $})
  • Elegir un estado I
    • Para cada item A -> α ⋅ X β ∈ I
      • Calcular goto(I, X).
      • Si goto(I, X) no es un estado ya existente, agregarlo como estado nuevo.
      • Agregar una arista etiquetada con X desde I hacia goto(I, X).
  • Repetir hasta que no se agreguen más estados.

LR(0)

Debilidades

  • Un item A -> α ⋅ implica una reducción.
  • Pero no indica cúando reducir, por lo que la aplicamos en todos los terminales.
  • ¿Qué sucede con los estados que contengan más de un item de esta forma?
  • Tendríamos un conflicto reduce/reduce.

  • ¿Qué sucede con los estados que contengan un item de esta forma, y además tengan transiciones salientes?
  • Tendríamos un conflicto shift/reduce.

SLR(1) parsing

Bottom-Up parsing

SLR(1)

  • Mejora sobre LR(0).
  • La idea es aprovechar la información del token de lookahead (1) para evitar conflictos.
  • Haremos una reducción sólo cuando el símbolo de entrada puede ir después del no terminal de la reducción.
  • Esto implica que debemos determinar qué terminaes pueden seguir a un no terminal.

Follow

SLR(1)

  • Para un no terminal A, Follow(A) es el conjunto de terminales que puede seguir A en alguna derivación.
  • $ ∈ Follow(S), si S es el símbolo de inicio.
  • Si A -> αBβ, entonces First(β) ⊆ Follow(B).
  • Si A -> αB, entonces Follow(A) ⊆ Follow(B).
  • Si A -> αBβ y β deriva ε, entonces Follow(A) ⊆ Follow(B).

LR(1) parsing

Bottom-Up parsing

LR(1)

  • Más completo que SLR(1).
  • Los items tendrán información de:
    • Producción.
    • Posición (el punto).
    • Símbolo de look ahead
  • Los items LR(1) tienen la forma [A -> α⋅B T]:
    • A -> αB es una producción.
    • ⋅ indica la posición.
    • T es un terminal o $.

LR(1)

  • Un item [A -> α⋅β T] significa:
    • El parser ya consumió α.
    • Si parsea β y el siguiente token es T
    • Entonces debe reducir usando A -> αβ.

Closure

LR(1)

Closure(I):
  • repeat
    • for each [A -> α ⋅ X β   c] ∈ I:
      • for each X -> γ
        • for each terminal d ∈ First(βc)
          • I = I ∪ {[X -> ⋅ γ   d]}
  • until no more items can be added to I, return I.

Goto

LR(1)

goto(I, X):
  • J = {}
    • for each [A -> α ⋅ X β   c] ∈ I:
      • J = J ∪ {[A -> α X ⋅ β   c]}
  • goto(I, X) = closure(J)

LALR(1) parsing

Bottom-Up parsing

LALR(1)

  • Simple mejora sobre LR(1).
  • LR(1) aumenta considerablemente la cantidad de items, por lo que puede llegar a tener muchos estados.

  • Si 2 estados son identicos, excepto por el token de look ahead de los items, se unen en un solo estado.
  • Esto reduce la cantidad de estados del DFA, mejorando el rendimiento del parser.

  • Nota: Es posible que esto provoque algunos conflictos reduce/reduce.
